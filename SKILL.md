# Multi-Format Build Skill

You are generating a build system that takes a standardized `source/manifest.json` and companion
markdown files, and compiles them into 6 distribution formats. The goal is to let the user write
their domain knowledge once and deploy it everywhere Гўв‚¬вЂќ in any language.

## When to use this skill

- User is creating a new skill or plugin project from scratch
- User wants to add multi-format output to an existing skill
- User asks for a build.ts, build pipeline, or scaffold for a skill project
- User mentions wanting their skill to work across Claude, OpenAI, n8n, MCP, or CLI
- User wants to localize an existing skill for additional languages
- User mentions i18n, locale, translation, or multi-language support for a skill

## Step 1: Establish or validate the manifest

Read `references/manifest-schema.md` for the full manifest JSON schema. The manifest uses a
`metadata` wrapper for project info, and JSON Schema-style `parameters` objects for commands.

If the user already has a manifest, validate it against the schema. If they have an older format
(flat fields, `CommandParameter[]` arrays instead of JSON Schema properties), migrate it to the
standard schema.

If the user is starting fresh, interview them about:
- Project name, description, author
- What skills (knowledge areas) the project covers
- What commands (user-facing actions) it exposes
- Parameter definitions for each command
- **What languages/locales the skill should support** (if any beyond the default)

Then generate the `source/manifest.json`.

### Locale configuration in the manifest

If the user needs multi-language support, add these fields to `metadata`:

```json
{
  "metadata": {
    "name": "Human-Readable Skill Name",
    "defaultLocale": "en",
    "locales": ["en", "es", "fr"],
    ...
  }
}
```

Rules:
- `defaultLocale` identifies which locale the base markdown files are written in
- `locales` lists all supported locale codes (BCP 47 format: `"en"`, `"es"`, `"pt-BR"`)
- `locales` must include `defaultLocale`
- When `locales` is omitted, the build operates in single-locale mode (backward compatible)

## Step 2: Set up locale files (multi-language only)

If the project supports multiple locales, create the locale infrastructure:

### 2a: Locale override files

For each non-default locale, create `source/locales/<locale>.json` containing translated
display strings. Read `references/manifest-schema.md` for the full locale override schema.

Key rules for locale override files:
- Only override human-readable strings (names, descriptions, displayNames)
- Do NOT override structural fields (parameter names, types, slugs, paths, keywords)
- Missing fields fall back to the default locale value from the manifest
- Parameter descriptions can be overridden per-command

### 2b: Localized markdown content

For each non-default locale, create subdirectories under `skills/`, `commands/`, and
`templates/`:

```
source/skills/es/<skill-name>.md
source/commands/es/<command-name>.md
source/templates/es/<template-name>.md
```

These contain the full translated domain content. The fallback chain is:
1. `source/skills/<locale>/<name>.md` Гўв‚¬вЂќ locale-specific version
2. `source/skills/<name>.md` Гўв‚¬вЂќ default locale fallback

This means you can incrementally translate: any missing locale file automatically falls back
to the default locale content.

## Step 3: Generate the build.ts

Read `references/build-template.md` for the complete build.ts template. This is the core of the
skill Гўв‚¬вЂќ a domain-agnostic TypeScript build script that reads the manifest and generates all 6
output formats, with full locale support.

The template is **fully manifest-driven with zero placeholder tokens**. Do NOT insert any
project-specific strings into the build.ts. Every value (name, slug, descriptions) is read
from `source/manifest.json` at build time. Locale resolution is handled automatically.

The build.ts handles:
- Manifest loading and validation (including locale configuration validation)
- Locale override merging and content fallback resolution
- A simple YAML stringifier (no external deps)
- CLI flags: `--format`, `--locale`, `--watch`, `--clean`, `--validate-only`
- Per-format and per-locale npm scripts in package.json

**Important**: The build.ts must be domain-agnostic. It reads the manifest dynamically Гўв‚¬вЂќ it should
never contain hardcoded skill names, command names, or domain-specific strings. Everything comes
from the manifest at build time. This includes the CLI generator: the CLI's help text and banner
must be constructed from `manifest.metadata.name` at build time, never inlined as string literals.
Copy the build template verbatim Гўв‚¬вЂќ do not modify it to include project-specific values.

## The 6 output formats

Each format serves a different integration target:

1. **Claude Plugin** (`dist/[locale/]claude-plugin/`) Гўв‚¬вЂќ `plugin.json` + `skills/` + `commands/`
   directories with SKILL.md and command markdown files. Ready to install in Claude Code.
   In multi-locale mode, `plugin.json` includes a `locale` field.

2. **OpenAI Functions** (`dist/[locale/]openai/functions.json`) Гўв‚¬вЂќ Array of function calling tool
   definitions with JSON Schema parameters. Works with GPT-4, GPT-4 Turbo, and compatible models.
   Descriptions are localized; function names stay in English.

3. **n8n Node** (`dist/[locale/]n8n/<Name>.node.json`) Гўв‚¬вЂќ Community node definition with operations
   mapped from commands and parameter fields with display options. `displayName` and operation
   names are localized.

4. **Prompt Library** (`dist/[locale/]prompts/`) Гўв‚¬вЂќ `index.yaml` + per-command YAML files with
   context, parameters, and output format specs. Model-agnostic prompt templates. Context role
   and descriptions are localized to guide LLM responses in the target language.

5. **MCP Server** (`dist/[locale/]mcp-server/`) Гўв‚¬вЂќ Full TypeScript MCP server project with tool
   definitions, knowledge loader, package.json, and tsconfig. Uses `@modelcontextprotocol/sdk`.
   Tool descriptions are localized; knowledge files use locale-specific markdown.

6. **Standalone CLI** (`dist/[locale/]cli/`) Гўв‚¬вЂќ A self-contained TypeScript CLI tool that exposes
   each command as a subcommand with argument parsing. Help text and banner are localized.

## Migrating existing projects

If the user has an existing skill project with a different manifest format (e.g., flat fields,
`CommandParameter[]` arrays, or missing the `metadata` wrapper), help them migrate:

1. Wrap top-level fields in a `metadata` object
2. Convert `CommandParameter[]` arrays to JSON Schema `properties` objects
3. Add `displayName`, `path`, and `keywords` fields to skills and commands
4. Add a `templates` array if templates exist
5. Validate the migrated manifest

### Adding locale support to an existing single-locale project

1. Add `defaultLocale` and `locales` to `metadata`
2. Create `source/locales/` directory
3. For each new locale, create `<locale>.json` with translated display strings
4. Create locale subdirectories under `skills/`, `commands/`, `templates/`
5. Add translated markdown content files
6. Add `build:locale:<code>` scripts to package.json for convenience
7. Run `npm run build` Гўв‚¬вЂќ the build system auto-detects locales from the manifest
