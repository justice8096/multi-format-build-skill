# Multi-Format Build Skill

You are generating a build system that takes a standardized `source/manifest.json` and companion
markdown files, and compiles them into 6 distribution formats. The goal is to let the user write
their domain knowledge once and deploy it everywhere.

## When to use this skill

- User is creating a new skill or plugin project from scratch
- User wants to add multi-format output to an existing skill
- User asks for a build.ts, build pipeline, or scaffold for a skill project
- User mentions wanting their skill to work across Claude, OpenAI, n8n, MCP, or CLI

## Step 1: Establish or validate the manifest

Read `references/manifest-schema.md` for the full manifest JSON schema. The manifest uses a
`metadata` wrapper for project info, and JSON Schema-style `parameters` objects for commands.

If the user already has a manifest, validate it against the schema. If they have an older format
(flat fields, `CommandParameter[]` arrays instead of JSON Schema properties), migrate it to the
standard schema.

If the user is starting fresh, interview them about:
- Project name, description, author
- What skills (knowledge areas) the project covers
- What commands (user-facing actions) it exposes
- Parameter definitions for each command

Then generate the `source/manifest.json`.

## Step 2: Generate the build.ts

Read `references/build-template.md` for the complete build.ts template. This is the core of the
skill â€” a domain-agnostic TypeScript build script that reads the manifest and generates all 6
output formats.

The template is **fully manifest-driven with zero placeholder tokens**. Do NOT insert any
project-specific strings into the build.ts. Every value (name, slug, descriptions) is read
from `source/manifest.json` at build time.

The build.ts handles:
- Manifest loading and validation
- A simple YAML stringifier (no external deps)
- CLI flags: `--format`, `--watch`, `--clean`, `--validate-only`
- Per-format npm scripts in package.json

**Important**: The build.ts must be domain-agnostic. It reads the manifest dynamically â€” it should
never contain hardcoded skill names, command names, or domain-specific strings. Everything comes
from the manifest at build time. This includes the CLI generator: the CLI's help text and banner
must be constructed from `manifest.metadata.name` at build time, never inlined as string literals.
Copy the build template verbatim â€” do not modify it to include project-specific values.

After generating all files, run:

```bash

```

Check that all 6 dist directories were created and contain the expected files.

## The 6 output formats

Each format serves a different integration target:

1. **Claude Plugin** (`dist/claude-plugin/`) â€” `plugin.json` + `skills/` + `commands/` directories
   with SKILL.md and command markdown files. Ready to install in Claude Code.

2. **OpenAI Functions** (`dist/openai/functions.json`) â€” Array of function calling tool definitions
   with JSON Schema parameters. Works with GPT-4, GPT-4 Turbo, and compatible models.

3. **n8n Node** (`dist/n8n/<Name>.node.json`) â€” Community node definition with operations mapped
   from commands and parameter fields with display options.

4. **Prompt Library** (`dist/prompts/`) â€” `index.yaml` + per-command YAML files with context,
   parameters, and output format specs. Model-agnostic prompt templates.

5. **MCP Server** (`dist/mcp-server/`) â€” Full TypeScript MCP server project with tool definitions,
   knowledge loader, package.json, and tsconfig. Uses `@modelcontextprotocol/sdk`.

6. **Standalone CLI** (`dist/cli/`) â€” A self-contained TypeScript CLI tool that exposes each
   command as a subcommand with argument parsing. Runnable via `tsx` or compiled.

## Migrating existing projects

If the user has an existing skill project with a different manifest format (e.g., flat fields,
`CommandParameter[]` arrays, or missing the `metadata` wrapper), help them migrate:

1. Wrap top-level fields in a `metadata` object
2. Convert `CommandParameter[]` arrays to JSON Schema `properties` objects
3. Add `displayName`, `path`, and `keywords` fields to skills and commands
4. Add a `templates` array if templates exist
5. Validate the migrated manifest
